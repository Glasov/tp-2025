# Лаба 5 - Agile и Scrum

На лекции мы разобрали Agile-подход, Scrum и роль процесса в командной разработке.
В этой лабораторной работе нужно смоделировать Scrum-процесс с использованием GitHub Issues.

Цель работы - понять, как:

* формулировать требования в виде задач,
* разбивать работу на инкременты,
* планировать спринт,
* отслеживать прогресс и завершённость.

## Что нужно сделать

1. Создать репозиторий для лабораторной работы
2. Настроить работу через GitHub Issues
3. Смоделировать один спринт по Scrum
4. Описать результат в README.md

Сдать ссылку на репозиторий.

## Содержание

### 1. Подготовка репозитория

Создать репозиторий.

В репозитории должен быть файл `README.md`, в котором кратко описан проект. Можно переиспользовать тему вашего проекта.

Описание должно занимать не более 5–7 предложений.

### 2. Product Backlog (Issues)

Создать беклог в виде GitHub Issues.

Требования:

* не менее 8 issues
* каждый issue описывает одну функциональную или нефункциональную задачу
* каждый issue содержит:
  * краткий заголовок
  * описание задачи (2–5 предложений)
  * критерии готовности (definition of done) в виде списка

### 3. Sprint Planning

Смоделировать один спринт.

Для этого:

1. Выбрать 4–5 issues из беклога
2. Отметить их как задачи текущего спринта:

   * добавить label `sprint-1`
   * добавить label `in-progress` / `todo` (на выбор)

В описании каждого issue, попавшего в спринт, указать:

* цель задачи,
* ожидаемый результат к концу спринта.

### 4. Scrum-артефакты в Issues

Для задач спринта:

* добавить оценку сложности (например, `story points: 1 / 2 / 3 / 5`)
  * в виде label или в тексте issue
* добавить приоритет (`low / medium / high`)

Таким образом, каждый issue в спринте должен содержать:

* описание,
* definition of done,
* приоритет,
* оценку сложности,
* признак принадлежности к спринту.

### 5. Ход спринта

Смоделировать выполнение спринта:

* часть задач перевести в состояние `in progress`
* минимум 2 задачи закрыть (closed)
* минимум 1 задачу оставить незавершённой

### 6. Sprint Review и Retrospective

В `README.md` добавить два раздела:

**Sprint Review**

Кратко описать:

* какие задачи были закрыты,
* что было доставлено по итогу спринта,
* какие задачи не успели завершить.

**Sprint Retrospective**

Кратко ответить на вопросы:

* что прошло хорошо,
* что пошло не так,
* что можно улучшить в следующем спринте.
